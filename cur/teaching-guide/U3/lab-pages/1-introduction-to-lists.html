Programming Lab 1: Introduction to Lists

Computers manipulate data and return processed information to the user. Numbers, strings of characters (letters, numbers, symbols, etc.), and Boolean values (true, false) are some of the basic pieces of information. Often, these small pieces are arranged in more complex structures: lists, ordered sets of data.

A list is not just a collection (set) but also a collection in a particular order (this order is tracked by an index number for each item on the list). If it has any elements at all, it has a first element, a last element, and so on. A list can also be totally empty, with no elements; often that is the way we want to set up (initialize) a list before we add elements to it. The length of an empty list is 0.

Lists are powerful. Databases are lists of lists of lists. Many operations can be performed on data in a list:

  • Searching a list to see if it contains some element.
  • Modifying lists to add, delete, or replace items, or to insert an item in a specific spot.
  • Creating new lists that keep only certain items from a list you started with.
  • You can create new lists with more or fewer elements than the list you have.
  • You can pick an item of a list by position or characteristic or just randomly.
  • You can do something with each item of the list: you can process it in some way, or for each element of the list, you can display it in some way.

Students begin page 1, Introducion to Lists, by building a basic shopping list app that introduces the ideas that lists can store data and that programs can access and manipulate list contents. In its simplest form, the app allows the user to add items or clear the list. Students can add more features as they learn more about list processing. (In Lab 2, the same basic idea of maintaining a list is used in a contacts list app, in principle like the ones on phones. That app will be revisited and improved several times in the curriculum.)

On page 2, Exploring List Processing, students explore various list blocks by predicting and testing the outcome of various expressions and small scripts. Students get a preview of several important procedures that will be addressed later in more detail (for each, map, keep, and combine).

Page 3, Modeling Sentence Structures, is a project that focuses on using abstraction to build structures such as noun phrases out of random selections from lists of words of various categories. Sentence phrase blocks are combined to build simple and complex sentences.

Pacing:

  • The first page of this lab (Shopping List App) should be relatively quick. Student may finish it and work on the Take it Further questions within one period.
  • The two remaining pages (Exploring List Processing and Modeling Sentence Structures) may take one to two class periods.

    • Opener: After the first day with "Shopping List App," present a short script with an error showing (where the error has been caused by neglecting to initialize a list). Ask students: What might have caused the error? What is the proper way to address this issue? (This opener connects to "Shopping List App," where students first initialize a list.)
    • Discussion: You may wish to review different types of data in Snap!: strings, lists, Booleans, numbers, functions, etc. You might ask:
      • What is is the difference between a string and a list? (Students without prior programming experience may reply, "what's the same between a string and a list?" The answer is that they're both aggregates. A string is an aggregate whose items are restricted to single characters; a list is an aggregate whose items can be anything.)
      • What else is special about a list? (Use if students don't mention that each item in a list has an index number.)
      • What can you do with a list? Listen for both specific tasks (e.g. "add an item" or "order the items") and also for project ideas (e.g. "track earnings and expenses" or "organize sports data").
      • What kind of data types can be list items? Invite students to experiment with Snap! to answer this question and to consider contexts for which including various data types in a list would be useful. (You should keep in mind that the right answer to questions like this is "any data type can be used anywhere," if the question means where a user is permitted to put some value. But this shouldn't be a mantra for kids to memorize.)
      • Can you put a list in a list? What are some examples of a list within a list? (Students may not have examples just yet.) This is both a preview of the next lab, "Nesting Lists," and also an introduction to the idea that nesting computational objects is common in programming.
  • Page 2: Exploring List Processing.
    • This page asks students to build the blocks (rather than providing a link that has the blocks built already). This is intentional, so that students get the experience of finding these blocks in the palette and getting the logic of what kinds of blocks go inside of other blocks.

If you don't hear students making this distinction, consider a brief class discussion about it. Knowing what data type is reported is a big help when debugging.

      • What does the ... block do? (for the following higher-order list blocks: for each, map, keep, and combine) Students don't need to achieve mastery here. This is just a chance to spend a moment building some initial ideas about these functions before Lab 3: Three Key List Operations.

  • Page 3: Modeling Sentence Structures.
    • The page is designed to offer a fun mini-project in which the use of lists feels natural and empowering. Encourage students to be creative and build other kinds of sentences or even simple conversations.
    • This lesson relies on proficiency in the ideas of this class more than knowledge of English sentence structure, but if you have students who are more proficient with another language, you may encourage them to build simple and complex sentence builders in a language of their choice. Students could work with others or alone and then share. Be creative. For example, it might be beneficial to pair a student studying Spanish with a fluent Spanish speaker. Bear in mind, though, that English is by far the easiest language for this project because it has hardly any inflections (case, tense, mood, etc.).
    • As always, if students do interesting work (e.g. with another language or with the Take It Further excercises), consider inviting them to share with the class.
    • Discussion, you might ask: How do the sentence builder blocks demonstrate abstraction?

      Throughout this Unit, students will have to in order to access the blocks they need. This is required every time a student opens a new snap window. (Links to projects in the curriculum have been shared with the tools already imported.)
    • The words "item" and "element" have the same meaning. We favor "item" because that's the name of the block that examines an item in a list. Try to develop the "item" habit, but it's not a big deal.
    • In Snap!, the first item of a list is item number 1, not number 0 as in some other languages. We think this is clearly advantageous for learners: "the third item" and "item 3" should refer to the same thing. Zero-origin list indexing is a leftover from low-level languages that do arithmetic on array indices. Don't make a fuss about this, either, but if a kid thinks zero-origin is "right" you can point out that ordinals (first, second, third) and cardinals (1, 2, 3) should match.
    • We are encouraging functional programming style, which in this case means favoring the list reporters over the list commands. Make sure that students understand, for example, that all but first of makes a new list; the original list given as its input is not modified.
